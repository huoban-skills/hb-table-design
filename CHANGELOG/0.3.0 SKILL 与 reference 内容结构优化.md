# 0.3.0 SKILL 与 reference 内容结构优化

> 2026-07-01

重新梳理 SKILL 和 reference 的内容结构，让原则更易于被 AI 理解和执行：SKILL.md 砍掉冗余段落，reference 统一为"动作/信号/例子"的决策表格式，description 只保留路由所需信息。

## 摘要

1. 优化: SKILL.md 精简，砍掉职责边界、语法等冗余段落，合并重复内容，从 ~320 行降至 ~250 行
2. 优化: description 重写，只写"做什么 + 什么时候触发"，去掉流程定位和下游交接描述
3. 优化: reference 结构统一，两篇参考文档按同一叙事模式重写——开头说本文干什么 → 查目录/分类表 → 按规则逐条对照（动作/信号/例子）
4. 优化: 示例统一为订单管理场景（客户、客户跟进、订单、订单明细、收款单、收款明细）

---

## 详情

### 1. 优化: SKILL.md 精简

- **删除**：职责边界（做/不做表格）、语法段（与模板重复）、渲染图片段（与流程步骤重复）、HTML 导出询问（HTML 已是主输出，无需询问）
- **合并**：语法 + 完整模板 → ER 图模板（一处说清格式和示例）
- **输出定位**：明确文档面向"搭建者和 AI"，是施工说明而非使用介绍
- **字段清单备注**：从技术语言（"业务主键""状态字段，置顶"）改为业务语言（"唯一标识每笔订单""跟踪订单从下单到完成的进度"）

### 2. 优化: description 重写

之前的 description 包含流程定位（"流程的哪个阶段"）、下游交接（"作为 xxx 的输入"）等路由系统不需要的信息。

重写为：`将业务需求转化为表结构方案（ER 图 + 字段清单），输出设计方案供确认。当用户说"设计表格""设计表结构""设计字段"，或创建表格中发现表结构不明确时触发。`

原则：description 是路由用的，只回答"这个 skill 做什么"和"什么时候该调用它"。

### 3. 优化: reference 结构统一

两篇参考文档重写为统一的叙事模式：

**字段选型规则.md**：
- 加开头主线：`本文用于确认每个字段该选什么类型`
- 选型规则分三个层次：主键设计 → 类型选择 → 用关联字段扩展
- 表格列从 4 列（场景/信号/选型/例子）改为 3 列（动作/信号/例子）——第一列是明确的执行动作（"增加自增编号""用关联替代选项"），不再是含混的 A vs B 场景描述
- "关联 vs 选项"决策表合并到"用关联字段扩展"下，与其他关联规则共用一张表，下方追加详细判断表并用"怎么判断"衔接

**表分类与分组.md**：
- 开头从"先分类、再分组"改为讲清 WHY：`本文用于确认每张表的定位和归属模块`
- 分组部分从目标/组名规范/步骤三个小标题 → 一句话引导 + 动作/信号/例子表格，与字段选型规则同一格式

### 4. 优化: 示例统一

SKILL.md 中的 ER 图模板、流程图 JSON、字段清单示例全部换为订单管理场景：
- 6 张表：客户、客户跟进、订单、订单明细、收款单、收款明细
- 3 个分组：客户管理、订单管理、收款管理
- 含计算字段示例：订单明细.小计（数量 × 单价）、订单.订单金额（汇总明细）、收款单.收款总额（汇总明细）
