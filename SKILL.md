# hb-table-design

## Skill 定位

**目标**：把业务需求转化为可确认的表结构方案，作为 hb-table-build 建表的输入

**输入**：可以是模糊的需求描述（如"做个工单管理"），也可以是线下表单、截图、Excel 或其他系统的表单截图。模糊需求主动反问，明确业务对象、流转状态和关键属性；现有表单/截图分析字段含义和关联关系，识别业务对象后重新设计为结构化方案，不做字段搬运

**输出**：单文件 HTML（`表结构设计.html`），自上而下包含系统说明 + 业务流程图 + ER 图 + 表格清单 + 字段清单。**文档面向搭建者（在伙伴云里建表的人和 AI），是施工说明而非给终端用户的使用介绍**

## 依赖关系

- 设计原则 → `reference/表结构设计原则.md`
- 字段选型 → `reference/字段选型规则.md`
- 表分类与分组 → `reference/表分类与分组.md`
- 渲染 ER 图 → `scripts/plantuml_render.py`
- 渲染业务流程图 → `scripts/flow_render.py`
- 生成 HTML → `scripts/md_to_html.py`
- 下游建表 → `../hb-table-build/SKILL.md`（由用户确认后触发）

## 执行流程

```
1. 理解需求
   ├─ 模糊需求 → 反问明确业务对象、流转状态、关键属性
   └─ 有表单/Excel/截图 → 分析字段含义和关联，识别业务对象

2. 设计表结构（按 reference/表结构设计原则.md 三层原则）
   ├─ 表级：识别实体（单一职责）、确定主键（主键约束）
   ├─ 关系级：一对多分离、多对多解耦、关系实体化
   └─ 字段级：原子化、消除重复组、字段归属、类型选型、计算替代

3. 质量门校验
   ├─ 孤立表检测
   └─ 命名规范

4. 输出
   ├─ 写 flow.json → python3 scripts/flow_render.py flow.json flow.svg
   ├─ 写 er.puml → python3 scripts/plantuml_render.py er.puml er.png
   ├─ 写 表结构设计.md（系统说明 + 业务流程图 + ER 图 + 表格清单 + 字段清单）
   ├─ python3 scripts/md_to_html.py 表结构设计.md → 生成 表结构设计.html
   └─ open 输出目录 && open 表结构设计.html

5. 确认门 → 等用户确认
```

## ER 图字段规则

ER 图只放三类字段，完整字段在字段清单中展开：

| 类别 | 说明 | 例子 |
|------|------|------|
| **主键字段** | 唯一标识一条记录的字段，含复合主键 | 合同编号；跟进记录 = 所属客户 + 时间 + 跟进人 |
| **状态字段** | 跟踪实体生命周期的状态字段 | 合同状态、工单状态、审批状态 |
| **必填字段** | 业务上不可缺失的核心字段 | 客户名称、负责人、关联的父表字段 |

非主键、非状态字段、非必填的字段一律省略。

**字段数量上限**：每个实体最多展示 **10 个字段**。超出部分在最后一行写 `"...": "+N 个字段，详见字段清单"`。

## 业务流程图

流程图放在文档最上方，画的是业务/数据怎么流动，不是表关系。只画 **L1 级**主干流程，**不要分支**，按分组画——每个分组对应一个业务模块（与 ER 图的 `package` 一致）。

每个环节标注对应的表（`table` 单张 / `tables` 多张），让搭建者直接看到「业务环节 ↔ 数据表」映射：

```json
{
  "groups": [
    {"name": "客户管理", "steps": [
      {"text": "录入客户", "table": "客户"},
      {"text": "跟进转化", "tables": ["客户", "客户跟进"]}
    ]},
    {"name": "订单管理", "steps": [
      {"text": "创建订单", "tables": ["订单", "订单明细"]},
      {"text": "确认发货", "table": "订单"}
    ]},
    {"name": "收款管理", "steps": [
      {"text": "登记收款", "tables": ["收款单", "收款明细"]}
    ]}
  ]
}
```

```bash
python3 scripts/flow_render.py flow.json flow.svg
```

- 环节用业务动词短语，每个分组 **2~5 个**环节
- 表名须与 ER 图/字段清单中的表名一致

## ER 图模板

用 `json` 定义表，`package` 按模块分组，箭头从父表（一）指向子表（多）。ER 图只放主键、状态字段、必填字段：

```
@startuml

package "客户管理" {
  json 客户 {
    "客户编号": "自增编号",
    "客户名称": "单行文本",
    "客户状态": "选项（潜在 / 跟进中 / 已成交 / 已流失）",
    "负责人": "成员"
  }
  json 客户跟进 {
    "所属客户": "关联数据",
    "跟进方式": "选项（电话 / 拜访 / 微信 / 邮件）",
    "跟进内容": "多行文本",
    "跟进人": "成员"
  }
}

package "订单管理" {
  json 订单 {
    "订单编号": "自增编号",
    "所属客户": "关联数据",
    "订单状态": "选项（待确认 / 已确认 / 已发货 / 已完成 / 已取消）",
    "下单日期": "日期",
    "订单金额": "计算（汇总订单明细.小计）"
  }
  json 订单明细 {
    "所属订单": "关联数据",
    "商品名称": "单行文本",
    "数量": "数字",
    "单价": "数字",
    "小计": "计算（数量 × 单价）"
  }
}

package "收款管理" {
  json 收款单 {
    "收款编号": "自增编号",
    "所属客户": "关联数据",
    "收款状态": "选项（待收款 / 部分收款 / 已收齐）",
    "收款日期": "日期",
    "收款总额": "计算（汇总收款明细.本次收款金额）"
  }
  json 收款明细 {
    "所属收款单": "关联数据",
    "关联订单": "关联数据",
    "本次收款金额": "数字"
  }
}

客户 --> 客户跟进
客户 --> 订单
订单 --> 订单明细
客户 --> 收款单
收款单 --> 收款明细

@enduml
```

```bash
python3 scripts/plantuml_render.py er.puml er.png
```

## 输出要求

### 输出目录

在项目目录下创建文件夹 `{业务名称}-表格搭建方案`，文件结构：

```
{业务名称}-表格搭建方案/
  flow.svg        # 业务流程图
  flow.json       # 流程图源数据
  er.png          # ER 图
  er.puml         # ER 图源码
  表结构设计.md   # 内容源文件
  表结构设计.html # 主输出（自动生成，含交互功能）
```

### 字段清单格式

字段清单包含所有字段（含 ER 图中省略的），每张表用 Markdown 表格输出，列：**字段名、类型、必填、唯一、备注**。

规则：
- **表单说明是字段行**：分组标题以 `表单说明` 类型的字段行表示，插在对应字段前，必填和唯一均填 —
- 表名后括号标注类型：主数据 / 业务表 / 基础数据 / 配置表
- 模块顺序与 ER 图中 `package` 顺序一致，同模块内先主数据后业务表
- 字段排序：**状态字段置顶 → 主键字段 → 表单说明行 → 该组字段**
- 必填列：业务上不能为空填 ✅，否则填 —
- 唯一列：该字段值在表内不能重复填 ✅，否则填 —
- 备注列：有需要解释的情况才写，不需要则留空

```markdown
#### 订单（业务表）
| 字段名 | 类型 | 必填 | 唯一 | 备注 |
|--------|------|------|------|------|
| 订单状态 | 选项（待确认 / 已确认 / 已发货 / 已完成 / 已取消） | ✅ | — | 跟踪订单从下单到完成的进度 |
| 订单编号 | 自增编号 | ✅ | ✅ | 唯一标识每笔订单 |
| 所属客户 | 关联数据 | ✅ | — | 关联到客户表 |
| 基本信息 | 表单说明 | — | — | |
| 下单日期 | 日期 | ✅ | — | |
| 备注 | 多行文本 | — | — | |
```

### 文档内容结构

````markdown
# {业务名称}-表格搭建方案

## 系统说明

> 写给搭建者的施工概览。用方案/结构口吻，让搭建者快速明白"要搭哪些表、怎么组织、为什么这么设计"。

一段话交代业务背景与方案范围（覆盖哪几个模块、共几张表），再用 3~6 条按模块列出各模块包含的表和设计取舍。

- **模块名**：包含哪些表、各自定位与承担的数据；如有合并/拆分取舍在此说明
````
