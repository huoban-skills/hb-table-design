# hb-er-draw

## Skill 定位

**目标**：把业务需求转化为可确认的表结构方案，作为 hb-table-build 建表的输入

**输入**：可以是模糊的需求描述（如"做个工单管理"），也可以是线下表单、截图、Excel 或其他系统的表单截图。模糊需求主动反问，明确业务对象、流转状态和关键属性；现有表单/截图分析字段含义和关联关系，识别业务对象后重新设计为结构化方案，不做字段搬运

**输出**：表结构设计文档（`表结构设计.md`），包含 ER 图（PNG）+ 字段清单（MD 表格），格式见「输出要求」节

## 依赖关系

- 设计原则 → `reference/表结构设计原则.md`（三层十条 + 质量门）
- 字段选型 → `reference/字段选型规则.md`（伙伴云字段类型 + 选型判断）
- 表分类与分组 → `reference/表分类与分组.md`（4 类表 + 分组规则）
- 渲染 ER 图 → `scripts/plantuml_render.py`
- 下游建表 → `../hb-table-build/SKILL.md`（本 skill 不调用，由用户确认后触发）

## 职责边界

| 本 Skill 负责 | 本 Skill 不负责 |
|---|---|
| 理解业务需求、设计表结构 | 在系统中创建表格 |
| 字段类型选型、关联关系设计 | 写 hac 命令、拼字段 JSON |
| 出 ER 图和字段清单、等用户确认 | 处理建表接口错误 |

## 执行流程

```
1. 理解需求
   ├─ 模糊需求 → 反问明确业务对象、流转状态、关键属性
   └─ 有表单/Excel/截图 → 分析字段含义和关联，识别业务对象

2. 设计表结构（按 reference/表结构设计原则.md 三层原则）
   ├─ 表级：识别实体（单一职责）、确定主键（主键约束）
   ├─ 关系级：一对多分离、多对多解耦、关系实体化
   └─ 字段级：原子化、消除重复组、字段归属、类型选型、计算替代

3. 质量门校验
   ├─ 孤立表检测
   └─ 命名规范

4. 输出文档
   ├─ 写 er.puml → 渲染 er.png
   ├─ 写 表结构设计.md（ER 图 + 字段清单）
   └─ open 输出目录

5. 确认门 → 等用户确认
```

## ER 图字段规则

ER 图只放三类字段，完整字段在字段清单中展开：

| 类别       | 说明                | 例子                          |
| -------- | ----------------- | --------------------------- |
| **主键字段** | 唯一标识一条记录的字段，含复合主键 | 合同编号；跟进记录 = 所属客户 + 时间 + 跟进人 |
| **状态字段** | 跟踪实体生命周期的状态字段     | 合同状态、工单状态、审批状态              |
| **必填字段** | 业务上不可缺失的核心字段      | 客户名称、负责人、关联的父表字段            |

ER 图中没有主键、无状态字段、非必填的字段一律省略，放进字段清单即可。

**字段数量上限**：每个实体最多展示 **10 个字段**。超出部分在最后一行写：

```
"...": "+N 个字段，详见字段清单"
```

> 示例：一张表有 15 个字段，ER 图只展示状态字段、主键、关键必填共 10 个，最后一行写 `"...": "+5 个字段，详见字段清单"`。

## 语法

用 `json` 定义表，`package` 按模块分组，箭头从父表（一）指向子表（多）：

```
@startuml

package "客户管理" {
  json 客户 {
    "字段名": "类型"
  }
}

客户 --> 联系人
客户 --> 线索 : 产生

@enduml
```

ER 图只放主键、状态字段、必填字段：

## 渲染图片

ER 代码要渲染成图片让用户直观确认，不要只丢 PlantUML 代码。
脚本在本 skill 的 `scripts/plantuml_render.py`，走官方 PlantUML 服务器渲染，需联网。

```bash
# 1. 把 PlantUML 代码写入 .puml 文件

# 2. 渲染为图片（PNG 或 SVG）
python3 scripts/plantuml_render.py <输入.puml> <输出.png>
```

- 输出扩展名只能是 `.png` 或 `.svg`
- 脚本会自动注入统一样式（字体、配色、白底），源 `.puml` 不被改动
- 渲染成功后把图片展示给用户，等确认

## 输出要求

### 输出目录

在项目目录下创建文件夹，命名格式：`{业务名称}-表格搭建方案`，例如 `商务部-表格搭建方案`。
文件夹中包含三个文件：

```
{业务名称}-表格搭建方案/
  er.png          # 渲染后的 ER 图
  er.puml         # PlantUML 源码
  表结构设计.md   # 主文档
```

`表结构设计.md` 中 ER 图引用使用相对路径 `![ER 图](er.png)`，`.puml` 文件与图片同目录。

### 字段清单格式

字段清单包含所有字段（含 ER 图中省略的），每张表用 Markdown 表格输出，列：**字段名、类型、必填、唯一、备注**。

规则：
- **表单说明是字段行**：分组标题以 `表单说明` 类型的字段行表示，插在对应字段前，必填和唯一均填 —
- 表名后括号标注类型：主数据 / 业务表 / 基础数据 / 配置表
- 模块顺序与 ER 图中 `package` 顺序一致，同模块内先主数据后业务表
- 字段排序：**状态字段置顶 → 主键字段 → 表单说明行 → 该组字段**
- 必填列：业务上不能为空填 ✅，否则填 —
- 唯一列：该字段值在表内不能重复填 ✅，否则填 —
- 备注列：有需要解释的情况才写，不需要则留空

```markdown
#### 表名（类型）
| 字段名 | 类型 | 必填 | 唯一 | 备注 |
|--------|------|------|------|------|
| 项目状态 | 选项（...） | ✅ | — | 状态字段，置顶 |
| 歌名 | 单行文本 | ✅ | ✅ | 业务主键 |
| 基本信息 | 表单说明 | — | — | |
| 商务 | 成员 | ✅ | — | |
| 制作进度 | 表单说明 | — | — | |
| 编曲负责人 | 关联数据 | — | — | |
```
