# hb-table-design

## Skill 定位

**目标**：把业务需求转化为可确认的表结构方案，作为 hb-table-build 建表的输入

**输入**：可以是模糊的需求描述（如"做个工单管理"），也可以是线下表单、截图、Excel 或其他系统的表单截图。模糊需求主动反问，明确业务对象、流转状态和关键属性；现有表单/截图分析字段含义和关联关系，识别业务对象后重新设计为结构化方案，不做字段搬运

**输出**：表结构设计文档（`表结构设计.md`），自上而下包含系统说明 + 业务流程图（SVG）+ ER 图（PNG）+ 表格清单 + 字段清单（MD 表格），格式见「输出要求」节。**文档面向搭建者（在伙伴云里建表的人），是施工说明而非给终端用户的使用介绍**。md 完成后询问用户是否额外导出单文件 HTML（便于发给客户、阅读体验更好）

## 依赖关系

- 设计原则 → `reference/表结构设计原则.md`（三层十条 + 质量门）
- 字段选型 → `reference/字段选型规则.md`（伙伴云字段类型 + 选型判断）
- 表分类与分组 → `reference/表分类与分组.md`（4 类表 + 分组规则）
- 渲染 ER 图 → `scripts/plantuml_render.py`
- 渲染业务流程图 → `scripts/flow_render.py`（读 JSON 生成 SVG，不依赖 PlantUML/网络）
- 导出单文件 HTML → `scripts/md_to_html.py`
- 下游建表 → `../hb-table-build/SKILL.md`（本 skill 不调用，由用户确认后触发）

## 职责边界

| 本 Skill 负责 | 本 Skill 不负责 |
|---|---|
| 理解业务需求、设计表结构 | 在系统中创建表格 |
| 字段类型选型、关联关系设计 | 写 hac 命令、拼字段 JSON |
| 出 ER 图和字段清单、等用户确认 | 处理建表接口错误 |

## 执行流程

```
1. 理解需求
   ├─ 模糊需求 → 反问明确业务对象、流转状态、关键属性
   └─ 有表单/Excel/截图 → 分析字段含义和关联，识别业务对象

2. 设计表结构（按 reference/表结构设计原则.md 三层原则）
   ├─ 表级：识别实体（单一职责）、确定主键（主键约束）
   ├─ 关系级：一对多分离、多对多解耦、关系实体化
   └─ 字段级：原子化、消除重复组、字段归属、类型选型、计算替代

3. 质量门校验
   ├─ 孤立表检测
   └─ 命名规范

4. 输出文档
   ├─ 写 flow.json → 渲染 flow.svg（业务流程图）
   ├─ 写 er.puml → 渲染 er.png
   ├─ 写 表结构设计.md（系统说明 + 业务流程图 + ER 图 + 表格清单 + 字段清单）
   └─ open 输出目录

5. 询问是否导出 HTML → 是则用 md_to_html.py 生成单文件 HTML

6. 确认门 → 等用户确认
```

## ER 图字段规则

ER 图只放三类字段，完整字段在字段清单中展开：

| 类别       | 说明                | 例子                          |
| -------- | ----------------- | --------------------------- |
| **主键字段** | 唯一标识一条记录的字段，含复合主键 | 合同编号；跟进记录 = 所属客户 + 时间 + 跟进人 |
| **状态字段** | 跟踪实体生命周期的状态字段     | 合同状态、工单状态、审批状态              |
| **必填字段** | 业务上不可缺失的核心字段      | 客户名称、负责人、关联的父表字段            |

ER 图中没有主键、无状态字段、非必填的字段一律省略，放进字段清单即可。

**字段数量上限**：每个实体最多展示 **10 个字段**。超出部分在最后一行写：

```
"...": "+N 个字段，详见字段清单"
```

> 示例：一张表有 15 个字段，ER 图只展示状态字段、主键、关键必填共 10 个，最后一行写 `"...": "+5 个字段，详见字段清单"`。

## 业务流程图

流程图放在文档最上方，让客户先看懂"这套系统怎么跑"，再看表结构。**它画的是业务/数据怎么流动，不是表关系**。

只画 **L1 级**：最高层的主干流程，**不要分支**（无 if/else、无判断节点），把每个业务模块的关键环节按先后顺序串成一条主线。**按分组画**——每个分组对应一个业务模块（与 ER 图的 `package` 模块一致），组内是该模块的几个关键环节。

流程图不用 PlantUML（自动排版丑、依赖公共服务器易限流）。改用 `scripts/flow_render.py`，写一份 JSON 描述分组与环节，脚本生成干净的 `flow.svg`（圆角卡片 + 分组配色，纯本地、无网络）。

**每个环节标注对应的表**（`table` 单张 / `tables` 多张），卡片下行会显示 `▸ 表名`，让搭建者直接看到「业务环节 ↔ 数据表」映射。多个环节指向同一张表（如建联、签约都落「达人」表）能直观说明状态字段驱动的合并设计：

```json
{
  "groups": [
    {"name": "达人运营", "steps": [
      {"text": "建联达人", "table": "达人"},
      {"text": "达人签约", "table": "达人"},
      {"text": "数据监管", "table": "达人数据监管"}
    ]},
    {"name": "单曲制作", "steps": [
      {"text": "词曲采买 / Demo 匹配", "tables": ["Demo 采买", "词曲拓展记录"]},
      {"text": "立项单曲", "table": "单曲项目"},
      {"text": "制作（编曲→录音→分轨→混音）", "table": "单曲项目"},
      {"text": "成品上会", "table": "单曲项目"},
      {"text": "宣推上线", "table": "单曲项目"}
    ]}
  ]
}
```

```bash
python3 scripts/flow_render.py flow.json flow.svg
```

- 环节用业务动词短语，每个分组 **2~5 个**环节，太细的步骤合并成一步
- 每个环节尽量标注对应表，表名须与 ER 图/字段清单中的表名一致
- 不画分支判断，只保留主干顺序
- 渲染成 SVG 后展示给用户确认，确认方式与 ER 图相同

## 语法

用 `json` 定义表，`package` 按模块分组，箭头从父表（一）指向子表（多）：

```
@startuml

package "客户管理" {
  json 客户 {
    "字段名": "类型"
  }
}

客户 --> 联系人
客户 --> 线索 : 产生

@enduml
```

## 渲染图片

ER 代码要渲染成图片让用户直观确认，不要只丢 PlantUML 代码。
脚本在本 skill 的 `scripts/plantuml_render.py`，走官方 PlantUML 服务器渲染，需联网。

```bash
# 1. 把 PlantUML 代码写入 .puml 文件

# 2. 渲染为图片（PNG 或 SVG）
python3 scripts/plantuml_render.py <输入.puml> <输出.png>
```

- 输出扩展名只能是 `.png` 或 `.svg`
- 脚本会自动注入统一样式（字体、配色、白底），源 `.puml` 不被改动
- 渲染成功后把图片展示给用户，等确认

## 输出要求

### 输出目录

在项目目录下创建文件夹，命名格式：`{业务名称}-表格搭建方案`，例如 `商务部-表格搭建方案`。
文件夹中包含以下文件：

```
{业务名称}-表格搭建方案/
  flow.svg        # 业务流程图（flow_render.py 生成）
  flow.json       # 流程图源数据
  er.png          # 渲染后的 ER 图
  er.puml         # ER 图 PlantUML 源码
  表结构设计.md   # 主文档
  表结构设计.html # 单文件 HTML（仅当用户确认导出时生成）
```

`表结构设计.md` 中图片引用使用相对路径 `![业务流程图](flow.svg)`、`![ER 图](er.png)`，源文件与图片同目录。

### 字段清单格式

字段清单包含所有字段（含 ER 图中省略的），每张表用 Markdown 表格输出，列：**字段名、类型、必填、唯一、备注**。

规则：
- **表单说明是字段行**：分组标题以 `表单说明` 类型的字段行表示，插在对应字段前，必填和唯一均填 —
- 表名后括号标注类型：主数据 / 业务表 / 基础数据 / 配置表
- 模块顺序与 ER 图中 `package` 顺序一致，同模块内先主数据后业务表
- 字段排序：**状态字段置顶 → 主键字段 → 表单说明行 → 该组字段**
- 必填列：业务上不能为空填 ✅，否则填 —
- 唯一列：该字段值在表内不能重复填 ✅，否则填 —
- 备注列：有需要解释的情况才写，不需要则留空

```markdown
#### 表名（类型）
| 字段名 | 类型 | 必填 | 唯一 | 备注 |
|--------|------|------|------|------|
| 项目状态 | 选项（...） | ✅ | — | 状态字段，置顶 |
| 歌名 | 单行文本 | ✅ | ✅ | 业务主键 |
| 基本信息 | 表单说明 | — | — | |
| 商务 | 成员 | ✅ | — | |
| 制作进度 | 表单说明 | — | — | |
| 编曲负责人 | 关联数据 | — | — | |
```

**打开目录**

文档写入完成后，立即执行以下命令打开输出目录，让用户直接看到文件：

```bash
open {业务名称}-表格搭建方案
```

**HTML 导出询问**

打开目录后，以对话形式询问用户（不写入文档）：

> 方案已生成 Markdown 文档。是否需要再导出一份单文件 HTML？HTML 是独立的单个文件，可直接发给客户，图片已内嵌，阅读体验更好。

用户确认后，用脚本生成单文件 HTML（图片自动内嵌为 base64），并 open 出来：

```bash
python3 scripts/md_to_html.py {业务名称}-表格搭建方案/表结构设计.md
open {业务名称}-表格搭建方案/表结构设计.html
```

**确认门**

最后以对话形式发送给用户（不写入文档）：

已完成「{业务名称}」的表格搭建方案，请确认方案，如果您需要，下一步我可以调用创建表格的技能，将以上方案搭建至系统中。
